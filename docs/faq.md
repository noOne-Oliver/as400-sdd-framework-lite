# 常见问题 FAQ

## Spec 相关

### Q: Spec 文档要写多详细？

A: 以"能让 Copilot 正确生成代码"为标准。主要检查点：
- 业务规则是否完整？（校验、权限、事务）
- 数据库设计是否提供了文件定义路径？
- 程序结构是否描述清楚了？
- Copilot 注意事项是否明确？

如果 Copilot 生成的结果经常需要大幅修改，说明 Spec 不够详细。

### Q: Spec 模板太复杂了，能简化吗？

A: 可以。但要确保包含核心要素：
- 业务规则（必须）
- 涉及的文件和字段（必须）
- Copilot 注意事项（必须）

其他部分可以简化。建议先跑通一个试点，再根据反馈调整模板复杂度。

### Q: Team Leader 太忙没时间写 Spec 怎么办？

A: 可以用 [Spec 辅助 Prompt](../prompts/spec/new-feature.md) 生成草稿，Team Leader 只需确认和补充。

或者让 Team Leader 只写核心的：
- 业务规则
- 涉及文件
Copilot 注意事项可以让开发者根据理解补充。

## Copilot 使用相关

### Q: Copilot 生成的结果不理想怎么办？

A: 检查顺序：
1. **Spec 是否完整？** — 这是最常见原因
2. **是否提供了文件定义路径？** — Copilot 需要读取
3. **是否明确了禁止事项？** — 防止常见错误
4. **Prompt 是否太长？** — 精简试试

如果还是不行，考虑：
- 分步骤生成（先结构后实现）
- 提供参考代码让 Copilot 学习风格
- 记录问题反馈给 Team Leader

### Q: Copilot 总是生成不存在的字段怎么办？

A: 在 Spec 的"Copilot 生成规则"中明确禁止：

```markdown
// 禁止代码模式
// ❌ 不要使用不存在的字段
CHAIN (OHXXX: OHYYY) ORDHDR;  // OHXXX 可能不存在！
```

同时确保提供了正确的文件定义路径。

### Q: Copilot 生成的代码风格不一致怎么办？

A: 在 Spec 中指定参考代码：

```markdown
// 代码风格参考：/src/PGMORD_R.rpgle
```

同时在 [Prompt 库](../templates/prompt-library.md) 中维护团队代码风格指南。

### Q: 如何减少 Copilot 幻觉？

A: 核心是提供足够的上下文：
- 文件定义源码路径（必须）
- 参考代码路径（推荐）
- 明确禁止事项（推荐）
- 分步骤描述（推荐）

详见 [Copilot 幻觉专项检查](../templates/review-checklist.md#copilot-幻觉专项检查)

## Review 相关

### Q: Review 需要多长时间？

A: 按阶段：
- 基础检查：5 分钟
- 业务逻辑检查：10 分钟
- 安全与性能检查：5 分钟
- Copilot 专项：5 分钟

总计约 25 分钟。如果代码复杂可以延长。

### Q: 发现 Copilot 幻觉问题怎么办？

A: 记录到 Review 记录表的"Copilot 幻觉问题记录"部分：

| # | 问题描述 | 猜测原因 | 建议改进 |
|---|----------|----------|----------|
| 1 | CHAIN 用了不存在的键 | Spec 没提供文件定义 | 更新 Spec |

每周汇总反馈给 Team Leader，用于改进 Spec 模板。

### Q: Review 通过的标准是什么？

A: 分严重度：
- **高严重度**：必须修复（编译错误、业务逻辑错误、安全问题）
- **中严重度**：建议修复，但不阻塞合并
- **低严重度**：可选修复

合并条件：
- 没有高严重度问题
- 至少 1 人 Approve

## 流程相关

### Q: 小改动也需要走完整流程吗？

A: 不需要。可以根据改动大小调整：
- **简单改动**（< 1 小时）：可以跳过 Spec 环节，直接 Copilot 生成 + Review
- **中等改动**（1-4 小时）：完整流程
- **复杂改动**（> 4 小时）：建议先评审 Spec，再生成代码

### Q: 多个开发者同时开发同一功能怎么办？

A: 建议：
- 一个功能分配给一个开发者
- 不同子功能可以并行
- 通过 Git 分支管理
- 定期 sync Spec 变更

### Q: 如何衡量流程效果？

A: 跟踪指标：
- 平均开发时间（vs 以前）
- Copilot 生成代码修改率
- Review 发现问题数
- 线上 bug 率

建议试点阶段每周收集这些数据。

## 技术相关

### Q: 支持哪些 RPG 类型？

A: 当前模板主要针对：
- RPGLE（ILE）
- FREE 格式

传统 RPG（带 C-spec）暂不支持，建议转换后再使用。

### Q: 支持 SQL 嵌入式吗？

A: 支持，但在 Prompt 中需要特别说明：
- 使用 EXEC SQL 语法
- 注意 SQL 注入风险
- 推荐使用参数化查询

### Q: 如何处理多文件事务？

A: 在 Spec 的"事务规则"中明确：
- 哪些文件在同一个事务中
- 错误时的回滚策略
- 锁控制要求

## 其他

### Q: 团队成员不配合怎么办？

A: 建议：
- 先试点，展示效果
- 收集正面案例分享
- 把流程收益量化（节省时间、减少 bug）
- 领导支持

### Q: 如何培训新成员？

A: 建议：
1. 阅读 [入门指南](getting-started.md)（15 分钟）
2. 观摩一次完整流程（30 分钟）
3. 在指导下完成一次开发（2-4 小时）
4. 独立完成一次 Code Review（30 分钟）

### Q: 框架需要持续维护吗？

A: 需要。主要维护工作：
- 根据反馈更新 Spec 模板（持续）
- 根据 Copilot 表现更新 Prompt（持续）
- 定期回顾 Review 发现的问题（每周）
- 优化流程效率（每月）
