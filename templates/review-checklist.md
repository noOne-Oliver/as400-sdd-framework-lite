# Code Review 清单 v2.0

> 支持 Fix Format 模式：Review 分为「检查」和「修复」两个阶段

---

## 第一阶段：代码检查

### 1. 基础检查（5 分钟）

#### 编译相关
- [ ] 代码能通过编译（CRTPGM / CRTRPGMOD）
- [ ] 没有未定义的变量
- [ ] 文件引用与实际定义一致

#### 结构相关
- [ ] 使用 FREE 格式
- [ ] 有完整的 DCL 语句
- [ ] 程序入口正确（DCL-PROC 或 *PSSR）

---

**如果这里有问题，直接打回重做。**

---

### 2. 静态规则检查（5 分钟）

> 使用 `templates/static-checks.md` 进行检查

#### 命名规范
- [ ] 程序命名符合规范
- [ ] 文件命名符合规范
- [ ] 字段命名符合规范

#### 代码格式
- [ ] 缩进正确（2 空格）
- [ ] 注释风格正确
- [ ] END 关键字完整

#### RPG 特定规范
- [ ] 使用 FREE 格式
- [ ] 过程定义正确（DCL-PROC / DCL-PI）
- [ ] 错误处理完整

---

### 3. 业务逻辑检查（10 分钟）

#### 规则检查
- [ ] 符合 Spec 中的业务规则
- [ ] 校验逻辑完整（必填、值域、格式）
- [ ] 权限判断到位

#### 数据处理
- [ ] 字段引用正确（无幻觉）
- [ ] 字符处理正确（TRIM/去空格）
- [ ] 数值处理正确（精度/四舍五入）
- [ ] 日期处理正确（格式/计算）

#### 异常处理
- [ ] 文件操作有错误判断
- [ ] 异常情况有日志
- [ ] 事务控制正确（COMMIT/ROLLBACK）

---

### 4. 安全与性能（5 分钟）

#### 安全检查 ⚠️ 重要
- [ ] 无硬编码密码/密钥
- [ ] 无硬编码内部路径/IP
- [ ] SQL 操作有参数化
- [ ] 敏感数据有脱敏处理

#### 性能检查
- [ ] CHAIN/READE 使用正确的键
- [ ] 没有在循环内重复打开文件
- [ ] 大数据量操作有分批处理
- [ ] 锁等待时间合理

---

### 5. Copilot 专项检查 ⚠️

#### 幻觉检查
- [ ] 所有 CHAIN/READE 的键值字段在文件中确实存在
- [ ] 所有引用的字段名与文件定义完全一致
- [ ] 没有"看起来合理但实际不存在"的字段

#### 一致性检查
- [ ] 代码风格与团队规范一致
- [ ] 命名规范与现有代码一致
- [ ] 错误处理方式与现有代码一致

#### 完整性检查
- [ ] 完整流程（初始化→处理→结束）
- [ ] 所有分支都有处理（IF/ELSE 完整）
- [ ] 所有资源都释放（文件关闭）

---

## 第二阶段：Fix Format（自动修复）

### 可自动修复的问题

以下问题可使用 Copilot 自动修复，发现后直接生成修复代码：

| # | 问题类型 | 修复方式 | 状态 |
|---|----------|----------|------|
| 1 | 缩进不一致 | 重新格式化 | [ ] |
| 2 | 大小写不一致 | 统一关键字大写 | [ ] |
| 3 | 多余空格/换行 | 清理多余空白 | [ ] |
| 4 | 注释格式不规范 | 统一注释风格 | [ ] |
| 5 | END 关键字缺失 | 补充完整结构 | [ ] |

### Fix Format Prompt

```markdown
## 任务：Fix Format
对以下 RPGLE 代码进行格式修复，不要改变任何逻辑。

## 原始代码
```rpgle
[粘贴需要修复的代码]
```

## 修复规则
1. 缩进统一为 2 空格
2. 关键字大写（IF, DCL, END, etc）
3. 字符串统一使用单引号
4. 清理多余的空格和空行
5. 确保所有 IF/DO/FOR 有对应的 END

## 输出格式
```rpgle
[修复后的代码]
```
```

## GitHub PR Review 流程

1. **PR 创建时**
   - [ ] PR 标题格式：`[AS400-PGM] 程序名 - 功能简述`
   - [ ] Description 链接对应的 Spec 文档
   - [ ] Attach Review 清单
   - [ ] 如有 Fix Format，标注 `[Fix Format]`

2. **Reviewer 职责**
   - [ ] 第一阶段：逐项检查（5 + 5 + 10 + 5 + 5 分钟）
   - [ ] 第二阶段：Fix Format（如适用）
   - [ ] 在 PR Comment 中标注问题
   - [ ] 使用「Request changes」或「Approve」

3. **开发者修复**
   - [ ] 处理每条 Review 意见
   - [ ] Fix Format 问题可直接接受 Copilot 建议
   - [ ] 在回复中说明修复方式
   - [ ] 完成后 @Reviewer 确认

4. **合并条件**
   - [ ] 至少 1 人 Approve
   - [ ] 所有高严重度问题已修复
   - [ ] 所有「必须修复」的静态检查已通过
   - [ ] CI/CD 编译通过（如果有）

## Fix Format 使用指南

### 何时使用

- 代码格式不规范但逻辑正确时
- Review 时发现可自动修复的问题
- 开发者主动请求格式修复时

### 如何使用

1. 复制需要修复的代码片段
2. 使用「Fix Format Prompt」生成修复建议
3. 开发者确认并应用修复
4. Reviewer 验证修复结果

### 注意事项

- ⚠️ Fix Format 只修复格式，不修复逻辑
- ⚠️ 修复前建议备份原代码
- ⚠️ 修复后必须重新 Review 确保逻辑未被改变
- ⚠️ 某些复杂格式问题可能需要手动修复
