# Code Review 清单 v3.0

> 支持 Fixed Format（旧版 RPG）和 Free Format（现代 RPGLE）
> 在 Spec 中指定代码格式，Review 时按对应格式检查

---

## 代码格式判定

根据 Spec 中指定的格式进行检查：

- [ ] **Fixed Format** — 传统 RPG，基于列的写法
- [ ] **Free Format** — 现代 RPGLE，自由格式
- [ ] **混用** — 主程序 Free，子程序 Fixed

---

## 第一阶段：基础检查（5 分钟）

### 编译相关
- [ ] 代码能通过编译（CRTPGM / CRTRPGMOD）
- [ ] 没有未定义的变量
- [ ] 文件引用与实际定义一致

### 结构相关
- [ ] 程序结构完整（初始化 → 主处理 → 结束）
- [ ] 所有文件正确打开和关闭
- [ ] 资源正确释放

---

**如果这里有问题，直接打回重做。**

---

## 第二阶段：格式检查（5 分钟）

### Fixed Format 检查 ⚠️

> 如果 Spec 指定为 Fixed Format，执行以下检查

| 检查项 | 规范 | 位置 | 必须修复 |
|--------|------|------|----------|
| 列位置 | 代码必须在 1-80 列内 | 整行 | [ ] |
| 操作码位置 | 操作码必须在列 7-21 | 整行 | [ ] |
| C-spec | 逻辑行必须以 C 开头 | 整行 | [ ] |
| 续行符 | 超过 80 列用 * 续行 | 列 6 | [ ] |
| 注释 | 列 1 用 / 或 * | 列 1 | [ ] |
| 顺序号 | 列 1-5 可选，必须有序 | 列 1-5 | [ ] |

### Free Format 检查 ⚠️

> 如果 Spec 指定为 Free Format，执行以下检查

| 检查项 | 规范 | 必须修复 |
|--------|------|----------|
| 声明 | 必须有 `**FREE` 或 `/free` | [ ] |
| 关键字大写 | IF/ELSE/ENDIF/DCL 等大写 | [ ] |
| 结构完整 | IF 有 ENDIF，DO 有 ENDDO，FOR 有 ENDFOR | [ ] |
| 缩进 | 2 空格缩进（禁止 Tab） | [ ] |
| 引号 | 单引号 `'text'` | [ ] |
| 注释风格 | `// 注释` 或 `/* 注释 */` | [ ] |

### 混用模式检查

| 检查项 | 规范 | 必须修复 |
|--------|------|----------|
| 主程序 | 使用 Free Format（`**FREE`） | [ ] |
| 子程序 | Fixed Format 部分有清晰边界 | [ ] |
| 调用 | Fixed Format 子程序用 CALL | [ ] |

---

## 第三阶段：静态规则检查（5 分钟）

> 使用 `templates/static-checks.md` 进行检查

### 命名规范
- [ ] 程序命名符合规范
- [ ] 文件命名符合规范
- [ ] 字段命名符合规范

### 安全检查 ⚠️
- [ ] 无硬编码密码/密钥
- [ ] 无 SQL 注入风险
- [ ] 敏感数据有脱敏

---

## 第四阶段：业务逻辑检查（10 分钟）

### 规则检查
- [ ] 符合 Spec 中的业务规则
- [ ] 校验逻辑完整（必填、值域、格式）
- [ ] 权限判断到位

### 数据处理
- [ ] 字段引用正确（无幻觉）
- [ ] 字符处理正确（TRIM/去空格）
- [ ] 数值处理正确
- [ ] 日期处理正确

### 异常处理
- [ ] 文件操作有错误判断
- [ ] 异常情况有日志
- [ ] 事务控制正确（COMMIT/ROLLBACK）

---

## 第五阶段：安全与性能（5 分钟）

### 安全检查
- [ ] 无硬编码内部路径/IP
- [ ] SQL 操作有参数化
- [ ] 敏感数据有脱敏处理

### 性能检查
- [ ] CHAIN/READE 使用正确的键
- [ ] 没有在循环内重复打开文件
- [ ] 大数据量操作有分批处理
- [ ] 锁等待时间合理

---

## 第六阶段：Copilot 专项检查 ⚠️

### 幻觉检查
- [ ] 所有 CHAIN/READE 的键值字段在文件中确实存在
- [ ] 所有引用的字段名与文件定义完全一致
- [ ] 没有"看起来合理但实际不存在"的字段

### 一致性检查
- [ ] 代码风格与 Spec 指定的格式一致
- [ ] 命名规范与团队标准一致
- [ ] 错误处理方式与现有代码一致

## GitHub PR Review 流程

1. **PR 创建时**
   - [ ] PR 标题格式：`[AS400-PGM] 程序名 - 功能简述`
   - [ ] Description 链接对应的 Spec 文档
   - [ ] 注明代码格式：`[Fixed]` / `[Free]` / `[Mixed]`
   - [ ] Attach Review 清单

2. **Reviewer 职责**
   - [ ] 确认 Spec 中的代码格式
   - [ ] 按对应格式检查（Fixed 或 Free）
   - [ ] 在 PR Comment 中标注问题
   - [ ] 使用「Request changes」或「Approve」

3. **开发者修复**
   - [ ] 处理每条 Review 意见
   - [ ] 在回复中说明修复方式
   - [ ] 完成后 @Reviewer 确认

4. **合并条件**
   - [ ] 至少 1 人 Approve
   - [ ] 所有高严重度问题已修复
   - [ ] 格式检查全部通过（Fixed 或 Free）
   - [ ] CI/CD 编译通过（如果有）

## 格式检查速查

### Fixed Format 必须满足
- [ ] C-spec 格式（列 7-21 为操作码）
- [ ] 代码在 1-80 列内
- [ ] 续行符在列 6
- [ ] 注释在列 1

### Free Format 必须满足
- [ ] 有 `**FREE` 或 `/free` 声明
- [ ] 关键字大写
- [ ] 结构完整（IF/ENDIF 配对）
- [ ] 缩进 2 空格
- [ ] 单引号字符串
