# Code Review 清单

> Reviewer 使用。按顺序逐项检查。

---

## 第一阶段：基础检查（5 分钟）

### 编译相关
- [ ] 代码能通过编译（CRTPGM / CRTRPGMOD）
- [ ] 没有未定义的变量
- [ ] 文件引用与实际定义一致

### 结构相关
- [ ] 使用 FREE 格式
- [ ] 有完整的 DCL 语句
- [ ] 程序入口正确（DCL-PROC 或 *PSSR）

---

**如果这里有问题，直接打回重做。**

---

## 第二阶段：业务逻辑检查（10 分钟）

### 规则检查
- [ ] 符合 Spec 中的业务规则
- [ ] 校验逻辑完整（必填、值域、格式）
- [ ] 权限判断到位

### 数据处理
- [ ] 字段引用正确（无幻觉）
- [ ] 字符处理正确（TRIM/去空格）
- [ ] 数值处理正确（精度/四舍五入）
- [ ] 日期处理正确（格式/计算）

### 异常处理
- [ ] 文件操作有错误判断
- [ ] 异常情况有日志
- [ ] 事务控制正确（COMMIT/ROLLBACK）

---

**这里有问题需要讨论后修复。**

---

## 第三阶段：安全与性能（5 分钟）

### 安全检查 ⚠️ 重要
- [ ] 无硬编码密码/密钥
- [ ] 无硬编码内部路径/IP
- [ ] SQL 操作有参数化（如果用嵌入式 SQL）
- [ ] 敏感数据有脱敏处理

### 性能检查
- [ ] CHAIN/READE 使用正确的键
- [ ] 没有在循环内重复打开文件
- [ ] 大数据量操作有分批处理
- [ ] 锁等待时间合理

---

**这里有问题需要评估影响后决定是否修复。**

---

## 第四阶段：Copilot 专项检查 ⚠️ 容易出问题的地方

### 幻觉检查
- [ ] 所有 CHAIN/READE 的键值字段在文件中确实存在
- [ ] 所有引用的字段名与文件定义完全一致
- [ ] 没有"看起来合理但实际不存在"的字段

### 一致性检查
- [ ] 代码风格与团队规范一致
- [ ] 命名规范与现有代码一致
- [ ] 错误处理方式与现有代码一致

### 完整性检查
- [ ] 完整流程（初始化→处理→结束）
- [ ] 所有分支都有处理（IF/ELSE 完整）
- [ ] 所有资源都释放（文件关闭）

---

**发现 Copilot 幻觉问题 → 记录到反馈池，更新 Spec 模板**

## GitHub PR Review 流程

1. **PR 创建时**
   - [ ] PR 标题格式：`[AS400-PGM] 程序名 - 功能简述`
   - [ ] Description 链接对应的 Spec 文档
   - [ ] Attach Review 清单

2. **Reviewer 职责**
   - [ ] 用 Review 清单逐项检查
   - [ ] 在 PR Comment 中标注问题
   - [ ] 使用「Request changes」或「Approve」

3. **开发者修复**
   - [ ] 处理每条 Review 意见
   - [ ] 在回复中说明修复方式
   - [ ] 完成后 @Reviewer 确认

4. **合并条件**
   - [ ] 至少 1 人 Approve
   - [ ] 所有高严重度问题已修复
   - [ ] CI/CD 编译通过（如果有）
